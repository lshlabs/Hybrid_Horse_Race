# Junior Change Workflow (Personal Checklist)

작성일: 2026-02-22  
목적: 이 프로젝트를 주니어 포트폴리오 관점에서 일관되게 수정하기 위한 개인용 체크리스트/요약 템플릿

## 1. 수정 전 체크리스트 (Problem First)

아래 4개를 먼저 적고 시작한다.

1. 문제 정의

- 지금 무엇이 문제인가?
- 사용자/플레이어 입장에서 어떤 증상이 보이는가?

1. 원인 가설

- 왜 발생한다고 보는가? (1~3개)
- 로그/코드/재현 절차로 확인 가능한가?

1. 변경 범위

- 어떤 파일을 수정할지
- 어떤 파일은 이번 작업에서 건드리지 않을지

1. 완료 기준

- 무엇이 되면 끝난 것으로 볼지
- 수동 테스트/자동 테스트 기준은 무엇인지

## 2. 수정 중 체크리스트 (Readable Code First)

1. 함수명이 의도를 설명하는가?

- "무엇을 하는지"가 함수명만 봐도 보이는지 확인

1. 한 함수가 너무 많은 일을 하지 않는가?

- 네트워크 호출 + 상태 변경 + UI 연출 + 복구 로직이 한 곳에 섞여 있으면 분리 고려

1. 실패 경로가 보이는가?

- `catch`, early return, fallback 여부가 읽히는지 확인
- 실패 시 로그 메시지가 디버깅에 도움이 되는지 확인

1. 주석이 필요한 지점만 설명했는가?

- 복잡한 이유/제약만 설명
- 코드 자체를 그대로 읽어주는 주석은 피함

1. 기능 변경과 구조 변경을 섞지 않았는가?

- 가능하면 분리해서 진행 (회귀 원인 추적 쉽게)

## 3. 수정 후 체크리스트 (Verification First)

1. 재현 케이스 다시 확인

- 원래 문제 재현 절차로 다시 테스트했는가?

1. 빌드/검증 실행

- `npm run build --prefix frontend`
- `npm run build --prefix functions`
- `npm run contract:check` (영향 범위가 멀티플레이 경로면 필수)

1. 회귀 여부 확인

- 내가 고친 곳 주변 기능이 깨지지 않았는가?
- 상태 전환/페이지 이동/에러 처리 흐름 확인

1. 문서 반영

- 플랜 문서 상태 변경 (`TODO/DOING/DONE`)
- 실행 로그/결정사항/검증 결과 기록

## 4. 개인용 작업 요약 템플릿 (포트폴리오 설명용)

작업 제목:

- 예) `RaceScene next-set transition sync 로직 분리 (CQ-02)`

문제:

- 어떤 증상/불편/위험이 있었는지 2~4줄

원인:

- 확인된 원인(또는 가장 유력한 원인) 1~3개

해결:

- 어떤 파일에서 무엇을 바꿨는지
- 구조 개선인지 기능 수정인지 명확히 구분

검증:

- 실행한 명령/수동 테스트
- 결과 (`PASS/FAIL`)

배운 점:

- 다음에 같은 작업을 할 때 재사용할 기준 1~2개

## 5. 이 프로젝트에서의 적용 원칙 (고정)

1. 포트폴리오 관점 우선

- 과도한 최적화보다 설명 가능한 구조 우선

1. 서버/멀티플레이 핵심 로직은 작은 단계로 변경

- 한 번에 크게 바꾸지 않기

1. 튜닝값 변경은 근거와 함께 기록

- 감으로 바꿨더라도, 테스트 결과를 같이 남기기

1. "왜 이 코드가 필요한지"를 말할 수 없는 변경은 보류

- 코드 양보다 설명 가능성이 더 중요함
