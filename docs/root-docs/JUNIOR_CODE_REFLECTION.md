# 주니어 수준의 개발자가 작성한 코드란 무엇인가

작성일: 2026-02-22  
대상 프로젝트: `Hybrid_Horse_Race`

## 1. 왜 이 문서를 쓰는가

나는 주니어 개발자이고, 이 프로젝트는 개인적인 흥미로 시작했지만 포트폴리오로도 활용하고 싶다.

그래서 중요한 질문이 생긴다.

- "주니어답다"는 것이 단순히 미숙하다는 뜻인가?
- 아니면 "현재 수준에서 문제를 정확히 이해하고, 설명 가능하게 해결하는 코드"를 말하는가?

이 문서의 결론은 두 번째다.

## 2. 내 기준에서의 좋은 주니어 코드

좋은 주니어 코드는 다음 특징을 가진다.

1. 문제를 숨기지 않는다.
- 버그를 우회만 하지 않고, 어떤 문제가 있었는지 코드/문서/로그로 남긴다.

2. 해결 의도가 읽힌다.
- "왜 이렇게 고쳤는지"가 변수명, 함수명, 작은 주석, 로그 메시지에서 드러난다.

3. 과한 추상화보다 이해 가능한 구조를 선택한다.
- 미래의 모든 경우를 미리 일반화하지 않는다.
- 지금 필요한 범위에서 분리한다.

4. 실패를 고려한다.
- 네트워크 실패, 상태 불일치, 재시도, 누락 데이터 같은 현실 문제를 다룬다.

5. 검증 가능한 형태로 남긴다.
- 빌드/테스트/체크리스트/재현 시나리오가 있다.

## 3. 주니어 코드에 대한 흔한 오해

### 오해 1: "짧고 똑똑해 보이는 코드"가 좋은 코드다
아니다.

주니어 포트폴리오에서는 특히,
- 읽기 쉬움
- 디버깅 쉬움
- 설명 가능성
이 더 중요하다.

### 오해 2: 성능 최적화를 많이 하면 실력이 좋아 보인다
아니다.

성능 최적화는 중요하지만, 문제 정의 없이 들어가면 오버엔지니어링이 된다.
주니어 단계에서는 먼저 다음 순서가 더 중요하다.

1. 기능이 맞게 동작하는가
2. 상태가 일관적인가
3. 실패 시 복구 가능한가
4. 그 다음에 성능을 측정하고 최적화하는가

### 오해 3: 복잡한 아키텍처를 쓰면 수준 높아 보인다
아니다.

복잡한 구조는 설명을 어렵게 만들고, 작은 변경도 위험하게 만든다.
주니어 코드의 강점은 오히려
- 단순한 구조
- 명확한 책임
- 명시적인 흐름
에 있다.

## 4. 이 프로젝트에서 특히 중요했던 주니어다운 좋은 선택

이 프로젝트는 단순 UI가 아니라 멀티플레이 동기화, 서버/클라 상태 전환, 레이스 재생 품질 같은 복합 문제를 다룬다.

이런 프로젝트에서 주니어 포트폴리오로 좋은 평가를 받을 수 있는 포인트는 다음이다.

1. "문제의 종류"를 구분한 것
- 기능 버그
- 동기화 버그
- UX 문제
- 운영/테스트 문제
를 섞지 않고 나눠서 접근한 것

2. 플랜 문서로 작업을 추적한 것
- `PLAN.md`, `PLAN_MULTIPLAYER_CONTRACT.md` 같은 문서로
- 우선순위, 상태, 검증, 리스크를 남긴 것

3. 멀티플레이 계약(전제 조건)을 명확히 고정한 것
- 호스트/게스트 룰
- 룸 삭제 조건
- 공유 파이프라인
같은 제품 규칙을 먼저 정의한 것

4. "간헐적 버그"를 테스트/체크리스트로 고정하려고 한 것
- 사람이 감으로만 테스트하지 않고
- 재현 시나리오와 contract check를 만든 것

이건 주니어에게 매우 좋은 습관이다.

## 5. 내가 피해야 할 주니어 코드 패턴

1. 증상만 고치고 원인을 기록하지 않는 코드
- 나중에 같은 문제가 다시 생긴다.

2. 이름이 모호한 코드
- `temp`, `data`, `value`, `handler2` 같은 이름이 많아지면 설명력이 떨어진다.

3. 한 함수가 너무 많은 일을 하는 코드
- 특히 멀티플레이/게임 상태 전환에서는 디버깅이 어려워진다.

4. 실패 경로를 무시하는 코드
- `catch {}`로 덮어버리면 실서비스에서 원인을 찾기 어렵다.

5. "나중에 정리"를 계속 미루는 코드
- TODO는 필요하지만, 무엇을 왜 미뤘는지 적어야 한다.

## 6. 포트폴리오에서 더 좋아 보이는 코드 작성법 (주니어 기준)

### A. 함수 이름을 동작 + 의도로 짓기
좋은 예:
- `handleReadyNextSet`
- `fillMissingRoundResultsFromServer`
- `throwInvalidSetIndex`

이름만 봐도 역할이 보인다.

### B. 로그를 "디버깅용 메모"가 아니라 "사건 기록"처럼 남기기
좋은 로그는 보통 이런 정보를 가진다.
- 어떤 액션인지 (`action`)
- 누구/어디서 발생했는지 (`roomId`, `playerId`)
- 기대값 vs 실제값 (`requestedSetIndex`, `currentSetIndex`)

### C. 복구 로직은 "조용히" 넣지 말고 의도를 드러내기
예:
- 누락 라운드 결과가 있으면 재시도
- 재시도 횟수 제한
- 실패 시 에러 로그 출력

이 흐름은 포트폴리오 설명에 매우 좋다.

### D. 문서와 코드 상태를 맞추기
README가 오래된 상태면 코드가 좋아도 신뢰도가 떨어진다.

포트폴리오에서는 특히
- 실행 방법
- 현재 상태
- 알려진 한계
- 다음 계획
을 정리하는 것이 중요하다.

## 7. "주니어답다"와 "미성숙하다"의 차이

### 주니어답다 (좋은 의미)
- 아직 모르는 것이 있지만 질문이 정확하다
- 문제를 단계적으로 쪼갠다
- 검증하고 수정한다
- 문서/로그를 남긴다

### 미성숙하다 (피해야 함)
- 감으로 고친다
- 재현/검증 없이 "된 것 같다"로 끝낸다
- 설명을 못한다
- 복잡한 코드로 포장한다

즉, 주니어 포트폴리오에서 중요한 건 "완벽함"이 아니라 "성장 가능한 문제 해결 방식"이다.

## 8. 이 프로젝트에 앞으로 적용할 나의 코딩 원칙

1. 먼저 문제를 정의하고, 그 다음 코드를 고친다.
2. 성능 최적화는 계측 후에만 한다.
3. 복잡한 추상화보다 읽기 쉬운 흐름을 우선한다.
4. 실패/예외 경로를 코드와 로그에 남긴다.
5. 변경 후에는 최소 1개 이상의 검증(빌드/테스트/시나리오)을 수행한다.
6. 포트폴리오 문서(README/리뷰 문서)도 코드의 일부라고 생각한다.

## 9. 면접/포트폴리오에서 이렇게 설명할 수 있다

이 프로젝트에서 내가 보여줄 수 있는 강점은 다음이다.

- 단순 기능 구현이 아니라, 멀티플레이 상태 동기화 문제를 단계적으로 해결한 경험
- 간헐적 버그를 체크리스트와 계약 테스트로 고정한 경험
- 서버 권위/클라이언트 재생 구조를 이해하고 적용한 경험
- 문서와 코드 상태를 맞춰가며 유지보수한 경험

즉, "화려한 코드"보다 "문제를 끝까지 추적하고 개선하는 개발자"라는 점을 보여주는 것이 더 중요하다.

## 10. 한 줄 결론

주니어 수준의 좋은 코드는 "어려운 기술을 쓰는 코드"가 아니라, **문제를 명확히 이해하고, 설명 가능하게 해결하며, 다시 검증할 수 있게 남긴 코드**다.
